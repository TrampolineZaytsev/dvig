Контрольные вопросы перед выступлением — ответы по проекту «ДВИГ»

1. Карта стейкхолдеров

Да. Мы выделяем пять групп:

  • Нуждающиеся — молодые горожане без устойчивой «тусовки», приезжие, интроверты и социально тревожные. Им нужен безопасный вход в компанию и офлайн без давления dating-формата.
  • Пользователи — активные искатели досуга и те, кто в чате «организует вечер». Им нужен быстрый консенсус «куда идём» и один понятный аргумент вместо пяти вкладок.
  • Союзники — вузовские медиа, студенческие чаты, площадки досуга, открытый источник KudaGo. Им выгодны актуальные данные, дайджесты, рост вовлечённости без лишней ручной работы.
  • Сопротивленцы — закрытые площадки без API, спам-сети, те, кто монополизирует дистрибуцию билетов. Им не выгодна прозрачная афиша и новый канал привлечения аудитории.
  • Официальные лица — 152-ФЗ, правила VK/Telegram, нормы площадок. Им важны прозрачность рассылок, возрастные маркеры, отсутствие вводящей в заблуждение рекламы.

Косвенная ЦА (кинотеатры, антикафе, музеи) ложится на союзников и частично на «официальные лица» площадок.

2. Кого продукт исключает — и почему

Минимум одна группа, которую мы осознанно не закрываем на текущем этапе:

Люди без смартфона, без стабильного интернета или без цифровой грамотности (в том числе старший возраст вне сегмента 18–28). ДВИГ завязан на мобильный/веб-интерфейс, афишу из API и заявки в приложении — без этого вход в продукт практически невозможен. Это не «их вина», а наш слепой угол, если говорить «помогаем всем». Мы честно ограничиваемся молодёжью в крупных городах на старте и планируем упрощать UX, но не подменяем офлайн-социальную инфраструктуру для всех возрастов.

Дополнительно на MVP не ориентируемся на жителей городов вне зоны данных (сейчас СПб через KudaGo) и на пользователей, которым нужен мгновенный романтический результат через свайпы — это другая логика продукта.

3. Кто теряет от появления продукта

Да, такие группы есть, мы их не замалчиваем:

  • Редакторы/SMM, чья работа сводится к простому копипасту афиши — автоматизация и шаблоны смещают их роль в сторону кураторства и смысла, а не механического переноса ссылок.
  • Площадки и каналы, зарабатывающие на непрозрачной рекламе в афише — культура «источник + честная пометка» снижает их преимущество.
  • Dating-сервисы в части сценария «хочу компанию на концерт, а не свидание» — мы забираем не весь их рынок, а другой JTBD: «пойти не с кем» через общее дело.
  • Небольшие площадки без API — трафик может концентрироваться у тех, кто уже в агрегаторах вроде KudaGo.
  • Пользователи теоретически могут «подсесть» на бесконечные подборки — риск для внимания и тревожности; поэтому мы не обещаем «идеальный вечер алгоритмом» и не раздуваем пуши.

4. Как пользователь выходит из отношений с продуктом

Выход должен быть простым и обратимым на этапе данных:

  • В приложении ДВИГ (целевая модель): скрыть профиль, запросить экспорт данных, подать на удаление аккаунта (с периодом на отмену, например 30 дней); отписаться от рассылок; удалить заявки из активных подборок.
  • Что остаётся: личные данные и фото удаляются; заявки и чаты обезличиваются для истории модерации; жалобы и safety-события — ограниченный срок; партнёрская аналитика — только в агрегированном виде без идентификации.
  • В пилоте MVP (spb-events / демо): пользователь-редактор удаляет локальный конфиг, убирает токены, перестаёт вызывать команды; в Telegram-чате — выходит из чата или договаривается с админом об отключении дайджеста.
  • Оспорить последствия полностью нельзя (кто-то уже увидел пост), поэтому важны дисклеймеры, ручное одобрение рассылок на пилоте и режим «только факты с KudaGo без ИИ».

5. Четыре угла рамки «Инженерия 2.0» (слайд 06)

На слайде должны быть заполнены все четыре оси — кратко, что туда класть:

  • Технологии: spb-events (KudaGo + опционально LLM + Telegram + export) и веб-демо ДВИГ (Next.js, прокси к афише); разделение факт / сгенерированный текст; кэш, retry, секреты вне репозитория. Риски: падение API, галлюцинации LLM, утечка токена.
  • Рынок и экономика: TAM — городской досуг; SAM — молодёжь 18–28, СПб/МСК; SOM — пилоты с 1–2 редакциями/чатами. Монетизация: бесплатно для пользователя → комиссии и размещение у организаторов; сейчас CLI — доказать ценность канала без выручки.
  • Человек: центр — общение («куда идём») и JTBD «пойти не с кем»; взлом нормы — от «сам перелопачу интернет» к куратору с ответственностью; краевые сценарии — тревога, одиночество, стигма «ещё одно приложение для знакомств».
  • Экология: углеродовый след LLM (кэш, короткие промпты, режим без ИИ); смысловая экология — продукт ведёт в офлайн, а не удерживает в ленте; операционная «экологичнее ручной вёрстки» — только как гипотеза с замером.

6. Конкретный запрос к аудитории

Запрос должен быть одним и измеримым, например:

  • Пилот: подключить 1–2 студенческих медиа или чата в СПб на 2–4 недели — тест дайджестов и обратная связь «доверяем / не доверяем».
  • Менторство / экспертиза: связь с площадкой досуга или юристом по 152-ФЗ для политики данных перед реальными встречами.
  • Не просим «инвестиции в масштабирование» на этом этапе, если защита учебная — просим проверку гипотезы и контакты для пилота.

Формулировка: «Нам нужен пилот с редакцией или админом чата, чтобы замерить время на подборку и качество доверия к формату — и одна площадка/наставник по безопасности офлайн-встреч».

7. Сложность проекта и этапы развития

Сложность: социотехническая — не только код, но доверие, модерация, юридика, поле. Технически MVP скромный (сборка API + демо UI); продуктово сложнее — социальный слой (группы, заявки, safety) и B2B-сеть.

Этапы:

  1. Сейчас: метаисследование + spb-events + веб-демо (афиша KudaGo, мок заявок/safety).
  2. Ближайшее: пилот B2B2C с редакциями; доработка лендинга и страниц доверия; честные дисклеймеры.
  3. MVP ДВИГ: бэкенд, профили, реальные заявки и модерация, группы 5–15, верификация.
  4. Масштаб: Москва, мульти-источник афиши, партнёрский кабинет, мобильное приложение.

8. Речь не должна дублировать слайды

Совет для выступления: на слайдах — схемы, таблицы, тезисы; в устной речи — полевые сцены (Манеж, «Этажи», «Родина»), голос пользователя («перебрал афиши — всё равно не иду один»), одна фраза «мы не сводим людей, мы даём повод и компанию». Не зачитывать таблицу конкурентов — объяснить разрыв рынка примерами. Не читать буллеты про B2B — сказать один кейс «редактор тратит вечер на пять вкладок».

10. Матрица рисков (слайд 10) — минимум один социальный риск

Да. Примеры социальных рисков с митигацией:

Риск Митигация
Потеря доверия из-за ошибки ИИ в тексте дайджеста Маркировка «сгенерировано», дисклеймер, ручная вычитка в пилоте, режим «только KudaGo»
Стигма «ещё одно приложение для знакомств» Позиционирование «дело → люди», не афишируем романтику как цель
Тревога и FOMO от частых подборок Редкие согласованные дайджесты, не обещать «идеальный вечер»
Инцидент на офлайн-встрече Публичные места, группы от N, модератор, чек-ин, тревожная кнопка (roadmap)

Плюс технические, рыночные, регуляторные (152-ФЗ, спам) и командные (мало разработчиков).

11. Технологический стек (слайд 03) — с зависимостями

Продуктовое веб-демо:

  • Next.js 16 (App Router), React 19, TypeScript
  • Tailwind CSS 4, UI (shadcn/base-ui)
  • Деплой: Vercel (ориентация)

Данные:

  • KudaGo Public API v1.4 (СПб) — основной источник афиши; кэш ~5 мин на сервере
  • Зависимость: доступность и условия API KudaGo

MVP spb-events (инженерный слой):

  • Python, Typer + Rich
  • OpenAI / GigaChat (опционально) — смысловые тексты; зависимость от провайдера и ключей
  • Telegram Bot API — share; зависимость от ToS и токена
  • Конфиг: ~/.spb-events/, права 600, переменные окружения

Зависимости (явно назвать на слайде): KudaGo → афиша; LLM → пояснительный текст (не факты); Telegram → доставка; без собственной БД на текущем демо — заявки и профиль только в браузере (мок).
