## 1. Идея и миссия

### 1.1. Суть

**«ДВИГ»** помогает молодым людям в больших городах выходить из социальной изоляции и находить компанию для живого общения **не через классические dating-приложения**, а через **совместные занятия и мероприятия по интересам**.

### 1.2. Почему не «рыночные» знакомства

Классическое dating-приложение — **витрина анкет**: свайп смахивает целую личность. Интроверты и те, кто не умеет себя «продавать», проигрывают. ДВИГ строит среду вокруг **общего дела** — как в жизни: сначала контекст и активность, потом близость.

Романтика может случиться, но **не афишируется** как цель продукта.

### 1.3. Цифра как средство

Технологии нужны, чтобы **вывести людей в офлайн**, а не удержать в ленте. Интернет задумывался для сближения; ДВИГ использует его как мост к реальным встречам.

### 1.4. Позиционирование одной фразой

> **Дело → люди → офлайн**

| Мы | Не мы |
|----|--------|
| Совместное дело, малые группы (5–15), публичный формат | Свайпы и «собеседование на роль партнёра» |
| Путь: интерес → приятель → друг / партнёр | Массовый ивент-маркетплейс без социального слоя |
| Афиша + компания + безопасный контекст | Замена VK/Telegram для своего круга |
| Цифра ведёт в офлайн | Бесконечный скролл ради скролла |

## 2. Проблема и Jobs To Be Done

### 2.1. Типичная сцена

Пятница вечер. В stories — друзья в компаниях. Есть интересное событие (кино, лекция, настолки), но **идти одному некомфортно**, а позвать **некого**. Афиша (KudaGo, VK) отвечает на «куда», но не на «**с кем**». Dating пробуют и удаляют — давление и фальшь.

**Ядро JTBD:** разрыв между «**есть событие**» и «**пойти не с кем**».

### 2.2. Второй слой боли (СПб, поле)

Даже с компанией мешает **хаос данных**: разное время на сайте площадки, в билетном приложении и в stories; решение «куда» — в чате из пяти вкладок.

### 2.3. Альтернативы и почему не закрывают боль

| Альтернатива | Что даёт | Почему не хватает |
|--------------|----------|-------------------|
| Dating (Twinby, Bumble, VK Dating) | Быстрый контакт | Давление свидания; интроверты проигрывают витрине |
| Афиши (KudaGo, Timepad) | Куда и когда | Не решают «с кем» |
| VK, Telegram | Связь со своими | Не ведут к новым связям в реале |
| Пойти одному | Случайность | Стресс для тревожных |
| Ручной копипаст афиши | Контроль | Время, ошибки, нет единого источника |

**Вывод:** рынок раздвоен — «токсичные быстрые знакомства» vs «афиша без социализации». ДВИГ — ниша посередине.

### 2.4. Job Stories

- **Компания:** событие есть, друзей с тем же интересом нет → группа → пойти вместе.
- **Естественное знакомство:** устал от свайпов → встреча за общим делом в группе.
- **Безопасность:** встреча из интернета → модератор, публичное место, чек-ин.
- **Партнёр (B2B):** пустые слоты в будни → организованные группы молодых людей.

### 2.5. Прогресс пользователя (Jobs-as-Progress)

| Направление | От → К |
|-------------|--------|
| Эмоциональный | Одиночество, тревога → принадлежность |
| Социальный | Незнакомец → приятель → друг / партнёр |
| Безопасность | Тревога → защищённость (группа, публичное место) |
| Экзистенциальный | Рутина → «автор» своей социальной жизни |

## 3. Целевая аудитория

### 3.1. Прямая ЦА (B2C)

| Параметр | Значение |
|----------|----------|
| Возраст | 18–28 |
| География пилота | Санкт-Петербург (далее Москва) |
| Сегменты | Студенты, junior/middle специалисты, приезжие, интроверты, социально тревожные |
| Каналы | Смартфон, VK, Telegram, TikTok |

**Боли:** изоляция, «где найти своих», стресс dating, «куда и с кем», нет плавного социального лифта без флирта.

**Барьеры:** безопасность, стигма «ещё одно приложение для знакомств», сомнение в эффективности.

### 3.2. Персоны

**Артём, 24, Junior-разработчик (СПб)** — переехал, круг сузился. Хочет «как на парах в универе»: сначала дело, потом близость. Сценарий: настолки → антикафе → микрогруппа.

**Алина, 23, маркетолог (СПб)** — боится dating, хочет на авторское кино не одна. Сценарий: группа ~5, модератор, одобрение заявки, публичное место. **На первом этапе — не 1-на-1.**

### 3.3. Косвенная ЦА (B2B)

Кинотеатры, антикафе, квиз-румы, музеи — молодая аудитория, групповые визиты, платное размещение.

### 3.4. Кого продукт не охватывает (честно)

- Без смартфона / цифровой грамотности
- Вне 18–28 и вне крупных городов на старте
- Вне зоны KudaGo (сейчас СПб)
- Кому нужен мгновенный романтический результат через свайпы

## 4. Бизнес-модель

### 4.1. Принцип

**Не брать деньги с пользователя за живое общение.** Платит посредник (организатор, площадка): реклама, билеты, групповые визиты, заполнение непиковых слотов.

### 4.2. Источники выручки

| Источник | Содержание | Статус |
|----------|------------|--------|
| Freemium для пользователей | Бесплатно на старте | Действует в пилоте |
| Партнёрские комиссии | Билеты, бронирования | Roadmap |
| Платное размещение | «Рекомендовано», подборки | Roadmap |
| B2B2C для редакций (`spb-events`) | Экономия времени на дайджестах | Гипотеза, нужен замер |

### 4.3. Выгоды B2B-партнёров

- Заполнение непиковых слотов (группы 5–15 в будни)
- Рост среднего чека (групповые визиты, бар, материалы)
- Аудитория 18–28 с заявленными интересами
- Меньше рассинхрона даты/цены в каналах
- Имидж «живого» пространства

### 4.4. B2B2C гипотеза (редакции)

Экономия времени на постах в Telegram/VK. **Целевая метрика:** минуты на одно событие (ручной пост ~15–25 мин vs шаблон ~2–5 мин) — **только с замером**, не декларировать без цифр.

### 8.2. Слои приложения

| Слой | Технология | Назначение |
|------|------------|------------|
| UI | React 19, Tailwind 4, shadcn/base-ui | Лендинг, браузер событий, админка |
| API | Next.js Route Handlers | Auth, группы, заявки, safety, аналитика |
| Auth | bcryptjs + jose (JWT в cookie) | Сессии пользователей и модераторов |
| ORM | Prisma 5 | Модели User, Group, Application и др. |
| Внешние API | KudaGo, Telegram Bot API | Афиша и уведомления |

### 8.3. Поток данных: афиша + группы

1. Клиент запрашивает `/api/events` → сервер тянет KudaGo → `mapKudagoToDvigEvent()` нормализует в `DvigEvent`.
2. Параллельно клиент запрашивает `/api/groups` → реальные группы из БД.
3. Хук `usePilotData` / функция `mergeEventWithGroups()` **склеивает** событие KudaGo с группой по `kudagoEventId`.
4. Если группы нет — `hasRealGroup: false`, кнопка «Группа скоро откроется».
5. Старый детерминированный мок в `map.ts` (`mockSocialFromKudagoId`) **перекрывается** реальными данными при наличии группы.

### 8.4. Роли

| Роль | Права |
|------|-------|
| `USER` | Профиль, заявки, check-in, жалобы, feedback |
| `MODERATOR` | + создание групп, одобрение/отклонение заявок, `/admin` |
| `ADMIN` | То же, что модератор (зарезервировано) |

## 10. Модель данных (Prisma)

### 10.1. Сущности

| Модель | Назначение | Ключевые поля |
|--------|------------|---------------|
| `User` | Аккаунт | email, passwordHash, role |
| `Profile` | Профиль | displayName, city, interests (JSON string), trustedContact, onboardingDone |
| `Group` | Группа на событие | kudagoEventId, eventTitle, eventDate, capacity, meetingPoint, telegramLink, status |
| `Application` | Заявка в группу | userId, groupId, status (PENDING/APPROVED/REJECTED/CANCELLED) |
| `CheckIn` | Отметка на месте | userId, groupId, status |
| `Report` | Жалоба / тревога | type (COMPLAINT/PANIC), message |
| `Feedback` | Оценка после встречи | rating 1–5, comment |
| `WaitlistEntry` | Запись в пилот с лендинга | email, telegram, interests |
| `AnalyticsEvent` | Продуктовая аналитика | name, payload |

### 10.2. Связи (ER)

```
User 1──1 Profile
User 1──* Application
User 1──* Group (как moderator)
Group 1──* Application
Group 1──* CheckIn
Group 1──* Feedback
Application *──1 User, *──1 Group
```

### 10.3. Статусы

- **Group:** `OPEN`, `FULL`, `CLOSED`, `CANCELLED`
- **Application:** `PENDING`, `APPROVED`, `REJECTED`, `CANCELLED`

## 12. Пользовательские сценарии (как в продукте)

### 12.1. Новый пользователь

1. Лендинг → «Выбрать мероприятие» или waitlist.
2. `/app` → регистрация (согласие с `/privacy`).
3. Онбординг: имя, город, 1–3 интереса.
4. Выбор события с открытой группой → «Подать заявку».
5. Ожидание одобрения (до 24 ч).
6. После одобрения: точка встречи + ссылка на Telegram в профиле.
7. В день события: check-in в настройках `/app`.
8. После check-in: оценка 1–5.

### 12.2. Модератор

1. Вход `/admin` (роль MODERATOR).
2. Создание группы: `kudagoEventId` из URL события KudaGo, дата, вместимость, точка встречи.
3. Одобрение/отклонение заявок в очереди.
4. Мониторинг check-in rate и метрик.
5. Реакция на тревожные сигналы (Telegram, если настроен).

### 12.3. Критерии успеха пилота

- ≥ 10 check-in за период
- ≥ 60% одобренных доходят до события
- Медиана «меньше одиноко» T+7 ≥ 3/5

## 13. Безопасность и данные

### 13.1. Принцип

Безопасность = **забота и прозрачность**, не «алгоритм защитит». Для тревожной ЦА — группы, публичные места, модератор, чек-ин, возможность выйти.

### 13.2. Правила пилота

- Группы **5–7** человек, только **публичные места**
- **Не 1-на-1** на первом этапе
- Возраст **18+**
- Тревожная кнопка → модератор (+ доверенный контакт в профиле); при угрозе жизни — **112**
- Подробнее: [PILOT_REGULATIONS.md](./PILOT_REGULATIONS.md)

### 13.3. Данные

| Тип | Где хранится (пилот) | Примечание |
|-----|----------------------|------------|
| Профиль, заявки | SQLite/PostgreSQL на сервере | До удаления аккаунта |
| Афиша | KudaGo API + серверный кэш | ~5 мин |
| Жалобы, safety | БД + опц. Telegram | Ограниченный срок (политика) |
| Аналитика | `AnalyticsEvent` | Обезличенно где возможно |

Юридика: черновики [`/privacy`](src/app/privacy/page.tsx), [`/terms`](src/app/terms/page.tsx). Перед масштабом — консультация юриста по 152-ФЗ, хостинг в РФ.

## 23. Итоговая ценность

**ДВИГ** — доверительная экосистема:

- **Пользователи** — путь от одиночества к отношениям разных типов через общее дело, без витрины анкет.
- **Партнёры** — молодая аудитория и заполнение слотов.
- **Редакции** (B2B2C) — меньше хаоса в афише при честной подаче фактов.

На этапе пилота техническое ядро — **сборка**: KudaGo + реальный социальный слой с модерацией + прозрачные дисклеймеры. Moat строится на **доверии, группах и партнёрской сети**, не на одном API.
